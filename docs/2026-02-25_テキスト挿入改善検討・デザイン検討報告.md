# VoiceToTextInput 検討報告

**日付:** 2026-02-25  
**対象:** ① カーソル位置へのテキスト反映の改善 ② デザイン検討

---

## 第一部：カーソル位置へのテキスト反映の改善

### 現状の仕組み

- クリップボードにテキストをコピーし、`CGEvent` で ⌘V をシミュレート
- 録音開始時にフォアグラウンドだったアプリを記憶し、停止時にそのアプリをアクティブ化してから貼り付け
- Accessibility API による直接挿入は、Cursor / VS Code 等で動作しなかったため未使用

---

### 役割A：実装エンジニアの視点

**原因の候補**

1. **フォーカス・アクティベーションのタイミング**
   - 録音ウィンドウ表示でフォーカスが VoiceToTextInput に移っている可能性
   - 300ms の待機では、Electron 系アプリのフォーカス復帰が間に合わない可能性

2. **貼り付けイベントの送信先**
   - `CGEvent.post(tap: .cghidEventTap)` はシステム全体に送信
   - アクティブウィンドウが正しくないと、別アプリに貼り付けられる

3. **Electron / Cursor の挙動**
   - プログラムからの貼り付けを制限している可能性
   - セキュリティやサンドボックスによる制約の可能性

**改善案**

| 案 | 内容 | 期待効果 |
|----|------|----------|
| A1 | 待機時間を 300ms → 500〜800ms に延長 | フォーカス復帰の確率向上 |
| A2 | Accessibility API を再試行（AXValue 等でより広い要素を対象） | ネイティブアプリでの成功率向上 |
| A3 | クリップボード貼り付けを 2 回試行（失敗時リトライ） | 一時的な失敗の吸収 |
| A4 | 貼り付け前にフォーカス要素を AX で確認 | デバッグ・検証に有用 |

---

### 役割B：アプリ開発者（macOS）の視点

**技術的な制約**

- macOS のサンドボックス外アプリでも、他アプリへのテキスト挿入は制限が多い
- Cursor / VS Code は Electron ベースで、AX の要素構造が通常のネイティブアプリと異なる
- `kAXTextAreaRole` 等が Electron のエディタにマッチしない可能性

**改善の方向性**

1. **Accessibility API の拡張**
   - `AXWebArea`、`AXGroup` 等も対象に含める
   - フォーカス要素の親階層をたどり、編集可能な要素を探索する

2. **フォールバックの明確化**
   - 自動貼り付けが失敗した場合の案内を強化
   - 「クリップボードにコピー済み。⌘V で貼り付け」を常に表示する等

3. **ユーザー設定**
   - 「貼り付け遅延」を設定可能にする（例: 300ms / 500ms / 800ms）
   - 環境に応じて調整できるようにする

---

### 役割C：ユーザー体験（UX）の視点

**ユーザーが感じる問題**

- 「話した内容がその場に入力される」という期待と、実際の挙動のギャップ
- 手動で ⌘V を押す必要があると、音声入力のメリットが薄れる

**UX 面の改善案**

1. **フィードバックの強化**
   - 貼り付け成功/失敗を明確に通知（トースト、アイコン変化など）
   - 失敗時は「⌘V で貼り付け」を目立つ形で案内

2. **ワークフローの見直し**
   - 録音終了後、ユーザーが意図したアプリにフォーカスを戻してから貼り付けする流れを想定
   - 「貼り付け」ボタンをウィンドウ内に用意し、ユーザーがクリックしたタイミングで貼り付ける方式も検討

3. **代替手段の案内**
   - Alfred / Raycast 等のランチャー連携
   - 特定アプリ向けの拡張機能（VS Code 拡張など）の可能性を README 等で紹介

---

### 第一部 まとめ（テキスト挿入）

| 優先度 | 改善案 | 実装難易度 |
|--------|--------|------------|
| 高 | 待機時間の延長（500〜800ms）とリトライ | 低 |
| 高 | 貼り付け結果の明確なフィードバック | 中 |
| 中 | Accessibility API の対象拡張 | 高 |
| 中 | 貼り付け遅延のユーザー設定 | 低 |
| 低 | 手動貼り付けボタンの追加 | 中 |

**結論:** まずは待機時間の延長とリトライ、フィードバック強化を実施し、効果を確認するのが現実的。Accessibility API の拡張は、調査コストを考慮しつつ段階的に検討する。

## 第二部：デザイン検討

### 役割D：UI/UX デザイナーの視点

**現状の課題**

- 機能重視で、ビジュアルの一貫性やブランド感が弱い
- 録音中/待機中の状態が、色とテキストだけでは分かりにくい
- フローティングウィンドウとして、他アプリとの調和が取りづらい

**デザイン方針の提案**

1. **ミニマル・プロフェッショナル**
   - 余白を活かしたレイアウト
   - システムフォント（SF Pro）をベースに、読みやすさを優先
   - 背景を半透明にして、下のコンテンツが透けて見える

2. **状態の可視化**
   - 録音中: 赤いインジケーター＋軽いパルスアニメーション
   - 待機中: グレーのインジケーター＋控えめな表示
   - 認識中: テキストエリアに軽いハイライトや枠線で「入力中」を表現

3. **アクセシビリティ**
   - コントラスト比の確保（WCAG 2.1 AA）
   - フォントサイズの調整余地
   - 色覚多様性を考慮したインジケーター（色以外の手がかりも用意）

---

### 役割E：ビジュアルデザイナーの視点

**ビジュアルの方向性**

1. **ダークモード対応**
   - システムの外観に追従（Light / Dark）
   - ダークモード時は、背景を `Color(nsColor: .windowBackgroundColor)` 等で統一

2. **カラーパレット案**
   - **A案（ニュートラル）:** グレー基調、アクセントに青
   - **B案（温かみ）:** ベージュ/オフホワイト、アクセントにオレンジ
   - **C案（テック）:** ダークグレー背景、アクセントに緑（ターミナル風）

3. **アイコン・インジケーター**
   - 録音: マイクアイコン＋赤いリング
   - 待機: マイクアイコン（グレー）または「待機」テキスト
   - 角丸を 8〜12pt に統一し、モダンな印象に

---

### 役割F：インタラクションデザイナーの視点

**インタラクションの改善**

1. **アニメーション**
   - ウィンドウ表示: フェードイン＋軽いスケール
   - 録音開始: インジケーターのパルス
   - テキスト更新: スクロール時のスムーズな遷移（既存の実装を維持）

2. **フィードバック**
   - 録音開始/停止: 短いサウンドまたはハプティック（対応デバイスのみ）
   - 貼り付け成功: チェックマークや短いアニメーション
   - 貼り付け失敗: 警告アイコン＋「⌘V で貼り付け」の案内

3. **ウィンドウの挙動**
   - ドラッグで移動しやすくする
   - 画面端にスナップするオプション（将来検討）
   - 最小化ではなく、閉じるか非表示にする選択肢

---

### 役割G：プロダクトデザイナーの視点

**全体の一貫性**

1. **デザインシステム**
   - 余白: 8pt の倍数（8, 16, 24）
   - フォント: 見出し 15pt / 本文 13pt / キャプション 11pt
   - 角丸: 8pt で統一

2. **他アプリとの関係**
   - Notta / SuperWhisper 等の参考にしつつ、macOS ネイティブらしさを出す
   - メニューバーアプリとして、常駐時の負担感を抑える

3. **優先順位**
   - 第一: 可読性・状態の分かりやすさ
   - 第二: ミニマルで邪魔にならない見た目
   - 第三: ブランド性・独自性

---

### 第二部 まとめ（デザイン）

| 観点 | 推奨方針 |
|------|----------|
| 全体トーン | ミニマル・プロフェッショナル、余白を活かす |
| カラー | システム外観に追従、アクセントは控えめに |
| 状態表現 | 録音中はパルス、待機中はグレーで明確に区別 |
| アニメーション | 軽いフェード・スケール、過度にしない |
| アクセシビリティ | コントラスト・フォントサイズを確保 |

**デザイン案の優先実装候補**

1. ダークモード対応（システム外観の自動追従）
2. 録音インジケーターのパルスアニメーション
3. 角丸・余白の統一（8pt ベース）
4. 貼り付け成功/失敗のフィードバック強化
